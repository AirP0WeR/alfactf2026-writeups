## Условие

В ультрамодном коворкинге всё просто. Две зоны: «Нищета» и «Покой». Никаких каст, никакой дискриминации... только аппаратный модуль, который решает, в какую зону тебе идти.

На входе устройство с кнопками молча мигает и показывает какие-то хексы. Сотрудники уверяют, что это нормально, просто диагностика. Некоторые посетители пытаются читать их как сигналы судьбы.

Но иногда система ведёт себя странно. Людей внезапно пускает туда, куда их не должно пускать. Но жалоб никто не пишет — вдруг, это какая-то премиум-функция.

Поговаривают, кто-то видел, как один умелец заставил устройство вывести на экран `alfa{...}`.

Разберись, что на самом деле происходит внутри: `peace_6ce0fda.tar.gz`.

## Решение

### 1. Связь с задачей «Покой»

Задача — продолжение [«Покоя»](../pokoy/). Тот же архив `peace_6ce0fda.tar.gz`, тот же Verilog-нетлист FPGA, те же T-таблицы AES-128. Но цель противоположная: в «Покое» расшифровывали готовый шифротекст, а здесь нужно **заставить само устройство вывести `alfa{...}` на OLED**, найдя последовательность нажатий на T9-клавиатуре.

### 2. Два режима дисплея

Устройство переключается между двумя режимами отображения:

- **«Нищета» (обычный режим)** — на OLED показываются hex-значения регистра `_M_00325_` (диагностика, пользователь видит «случайные» байты).
- **«Покой» (скрытый режим)** — на OLED показывается ASCII-содержимое регистра `_M_00324_` как текст 4×16 символов.

Переключатель — сигнал `_00146_` в `peace.v`:

- `_00146_ = 1` → «Нищета» (hex-дисплей)
- `_00146_ = 0` → «Покой» (ASCII-дисплей)

### 3. Trigger-условие активации скрытого режима

Анализ `peace.v` (строки 52078–52092) даёт условие переключения:

```verilog
_00138_ = OR четырёх условий, проверяющих M_324[0][39:32] == 0x7B  // '{'
_00145_ = OR четырёх условий, проверяющих M_324[3][127:120] == 0x7D // '}'
_00146_ = _00138_ | _00145_
```

Скрытый режим активируется, когда:

- байт `M_324[0][39:32]` == `0x7B` (`{`)
- **И** байт `M_324[3][127:120]` == `0x7D` (`}`)

Это границы флага `alfa{...}`. Иначе говоря, чтобы устройство показало текст, само AES-преобразование первых четырёх набранных байт должно случайно дать `M_324[0]`, оканчивающийся на `alfa{`, а `M_324[3]` — на `}`.

### 4. T9-клавиатура и кодировка символов

Ввод — multi-tap T9 на 4×4-кнопочной клавиатуре:

- цифры: `1*1`…`0*1`
- буквы: `2*2 = 'a'`, `2*3 = 'b'`, `2*4 = 'c'`, `3*2 = 'd'`, `3*3 = 'e'`, `3*4 = 'f'`, …
- `#` — one-shot shift (заглавная следующая буква)
- скобки и пунктуация: `d*3 = '{'`, `d*4 = '}'`, и т. д.

Полная таблица — в [`solve/t9.py`](./solve/t9.py).

### 5. Прорыв — `M_324` зависит только от первых 4 байт ввода

Эмпирические тесты на симуляторе (cocotb + Icarus Verilog) показали:

- ввод `aaaa`, `aaaabbbb`, `aaaaaaaaaaaaaaaa` → одинаковое `M_324[0]`
- ввод `abcd`, `abcd1234`, `abcdefghijklmnop` → одинаковое `M_324[0]`

Регистр `_M_00324_` (тот, что показывается в скрытом режиме) зависит **только от первых 4 байт** введённого текста, а не от всех 64, как `_M_00325_`. Это сводит пространство поиска с 256⁶⁴ к 80⁴ ≈ 41 млн вариантов (символы, типизируемые на T9).

### 6. Ошибочная и правильная формула `M_324[0..3]`

Первая попытка использовала ту же AES-схему, что и в «Покое»: 16-байтовый блок ввода, прогоняется через 9 раундов + финальный, на выходе — `M_324[i]` после XOR с per-row маской из ROM `_M_00331_`. Сошлось **только** для строки 0. Строки 1, 2, 3 систематически отличались.

Корень — в ветке `_00793_[2] = 1` (скрытый режим) нетлист подаёт в AES не обычный 64-байтовый input register, а **специальный 16-байтовый блок**, в который зашит индекс строки:

```python
def pt_for_M324(row: int, b0, b1, b2, b3) -> bytes:
    # row попадает в старший байт через биты 120–121 (первый байт состояния)
    return bytes([row]) + bytes([0] * 11) + bytes([b3, b2, b1, b0])

M_324[row] = AES(pt_for_M324(row, M_160[0], M_160[1], M_160[2], M_160[3])) XOR mask[row]
```

Байт `row` влияет на старший байт состояния через биты 120 и 121. Поэтому блок 0 (`row=0`) сошёлся даже со старой формулой (нулевой байт ничего не меняет), а блоки 1..3 — нет. Фикс — в [`solve/m324_formula.py`](./solve/m324_formula.py); self-test для пустого ввода и `'1'` проходит для всех четырёх строк.

### 7. Ещё один подвох — порядок байт OLED

Старый отладочный код декодировал `M_324[i]` в обычном big-endian порядке. Реальный OLED-контроллер пишет 128-битную строку **LSB-first**: первый видимый символ — это `M_324[i][7:0]`, последний — `M_324[i][127:120]`.

Поэтому сравнение префикса с `'alfa{'` нужно делать по байтам `[15], [14], [13], [12], [11]` 16-байтового big-endian представления `M_324[0]`, и `'}'` — по байту `[0]` блока 3.

### 8. Векторизованный перебор

[`solve/niwhta_solver.py`](./solve/niwhta_solver.py) делает следующее:

1. Берёт `CHARSET` из 80 типизируемых на T9 символов.
2. Батчами по 1М перебирает 4-байтовые префиксы, целиком на NumPy:
   - строит 16-байтовое state с `row=0` и зашитыми 4 байтами префикса;
   - прогоняет 9 раундов AES через 144 T-таблицы из `peace.v`;
   - применяет финальный round (16 S-box'ов с `key_in`, `key_out`);
   - XOR'ит с per-row маской из `_M_00331_[0]`.
3. Фильтрует по байтам `[15..11] == "alfa{"`.
4. Для выживших проверяет `row=3`, байт `[0] == '}'`.

Полный поиск занимает несколько минут на ноутбуке. Единственный найденный ввод:

```text
y4am
```

T9-последовательность для набора:

```text
9*4 4*1 2*2 6*2
```

OLED-строки после подачи `y4am`:

```text
alfa{niwhta_or_p
okoy_decided_in_
silicon_6fb9b724
9ff149931f8b959}
```

### 9. Верификация на симуляторе

Финальная проверка через cocotb-симуляцию:

```bash
cd niwhta/solve
POKOY_TEXT=y4am ../../pokoy/solve/venv/bin/python3 visual_probe.py
```

Результат:

```text

row 1: 'okoy_decided_in_'

```

`screen_probe.py` дополнительно снял реальный OLED-снимок по байтам framebuffer'а и подтвердил совпадение.
