## Решение

### Анализ приложения

Приложение представляет собой платформу для игры в падель с мультиплеером в реальном времени на WebSocket. Основные компоненты:

1. **Система аутентификации** — регистрация/логин с сессионными куки (`padel_sid`)
2. **Система комнат** — обычные пользовательские комнаты и специальные "официальные" комнаты с ботами
3. **WebSocket API** — связь в реальном времени с различными типами сообщений (`c_*` клиентские, `s_*` серверные)
4. **Официальные комнаты** — специальные комнаты с 3 ботами, содержащие историю чата с флагом

### Обнаруженная уязвимость

Уязвимость заключается в механизме эскалации привилегий в обработчике WebSocket сообщений.

**Файл:** `server/GameServer.js:657`
```javascript
const { playerId: leavingId } = { playerId, ...msg };
```

Сервер использует деструктуризацию объекта с spread оператором, что позволяет клиентскому `msg.playerId` переопределить аутентифицированный параметр `playerId` в сообщениях `C_LEAVE_ROOM`. Это позволяет исключать других игроков (включая ботов), выдавая себя за них.

**Условие победы** в `server/Room.js:209-222`:
- Когда хост комнаты уходит, система переназначает `hostId` следующему игроку в Map
- Если новый хост находится в официальной комнате И является реальным игроком (не ботом) И имеет активное WebSocket соединение, система вызывает `room._replayChatHistoryTo(newHost.ws)`
- История чата официальной комнаты была заранее заполнена содержимым с флагом

### Стратегия эксплуатации

1. **Регистрация аккаунта** — создаём сессию и получаем куки аутентификации
2. **Поиск официальных комнат** — подключаемся к `/ws` и запрашиваем список официальных комнат
3. **Вступление в официальную комнату** — подключаемся к `/ws?room=<official_id>` для вступления в комнату с 3 ботами
4. **Эскалация до хоста** — использование уязвимости подмены playerId для последовательного исключения всех ботов:
   - Отправляем `{"type": "c_leave_room", "playerId": "<bot1_id>"}` → хостом становится bot2
   - Отправляем `{"type": "c_leave_room", "playerId": "<bot2_id>"}` → хостом становится bot3
   - Отправляем `{"type": "c_leave_room", "playerId": "<bot3_id>"}` → хостом становимся мы
5. **Получение повтора чата** — когда мы становимся хостом, сервер автоматически воспроизводит историю чата с флагом

### Выполнение эксплойта

```python
# Ключевые сообщения эксплойта
for bot_id in bot_ids:
    cmd = {"type": "c_leave_room", "playerId": bot_id}
    await ws.send(json.dumps(cmd))
```

Эксплойт работает потому что:
- Официальные комнаты имеют предопределённую историю чата с флагом: `"Он никогда не сможет выиграть у нас и получить этот флаг: alfa{wINNer_wiNNER_g0rPK0R5HiCK_dINner}"`
- Когда последний бот удаляется, наш игрок становится новым хостом
- Функция `_replayChatHistoryTo()` отправляет все исторические сообщения чата новому хосту
- Извлекаем флаг из повтора чата с помощью regex: `alfa{wINNer_wiNNER_g0rPK0R5HiCK_dINner}`

### Технические детали

Цепочка уязвимости:
1. **Приоритет object spread** — `{ playerId, ...msg }` позволяет `msg.playerId` переопределить аутентифицированный `playerId`
2. **Порядок итерации Map** — JavaScript Map сохраняет порядок вставки, делая последовательность хостов предсказуемой
3. **Условный повтор чата** — официальные комнаты с реальными игроками-хостами запускают автоматический повтор истории чата
4. **Предзаполненный контент** — официальные комнаты содержат захардкоженные сообщения чата, включая флаг

Эксплойт надёжен, поскольку удаление ботов и переназначение хоста происходит синхронно, а повтор чата запускается немедленно при становлении хостом официальной комнаты.
