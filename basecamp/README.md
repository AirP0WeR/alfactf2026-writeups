## Анализ системы

1. **Архитектура приложения**: BaseCamp — образовательная платформа на Golang с отдельным основным сервером и микросервисом отзыва токенов
2. **Модель VIP доступа**: Курс 4 имеет специальный VIP урок (#16), требующий OneTime JWT токен, получаемый через `/request-access`
3. **Критическая уязвимость**: Race condition deadlock в управлении goroutine'ами сервиса отзыва
4. **Поведение Auth Middleware**: Работает в режиме fail-open при ошибках сервиса отзыва, только логирует но не блокирует запросы
5. **Поток OneTime токенов**: VIP токены автоматически отзываются после однократного использования через after-request hook middleware

## Решение

### Первоначальная разведка

Приложение предоставляет исходный код, раскрывающий архитектуру микросервисов:
- Основной сервер обрабатывает курсы, уроки и аутентификацию
- Отдельный сервис отзыва управляет чёрным списком JWT токенов
- VIP урок #16 в курсе #4 содержит флаг, но требует специального OneTime JWT доступа

### Анализ уязвимости

**Коренная причина**: `revocation/internal/service.go` содержит критическое состояние гонки в методах `CheckRevoked()` и `Revoke()`:

```go
func (s *Service) CheckRevoked(jti string) (bool, error) {
    ch := make(chan bool)  // НЕбуферизованный канал
    
    go func() {
        s.store.mu.Lock()
        defer s.store.mu.Unlock()
        revoked := s.store.IsRevoked(jti)
        ch <- revoked  // Блокируется навсегда, если нет получателя
    }()
    
    select {
    case result := <-ch:
        return result, nil
    case <-time.After(50 * time.Millisecond):  // Срабатывает таймаут
        return false, fmt.Errorf("check timeout")
    }
}
```

**Проблема**: 
1. Когда файловый I/O занимает >50ms (большой файл отзывов), срабатывает таймаут select
2. Основная goroutine возвращается с ошибкой, но рабочая goroutine продолжает выполнение
3. Рабочий процесс завершает сканирование файла, удерживая блокировку `store.mu`
4. Рабочий процесс пытается `ch <- revoked`, но канал не имеет получателя (небуферизованный) → **блокируется навсегда**
5. `defer s.store.mu.Unlock()` **никогда не выполняется** → mutex навсегда заблокирован
6. Все последующие проверки отзыва завершаются таймаутом → auth middleware работает в режиме fail-open

**Поведение Fail-Open**: В `middleware/auth.go` ошибки отзыва только логируют предупреждения, но не блокируют запросы:

```go
revoked, err := revClient.Check(c.Request.Context(), claims.JTI)
if err != nil {
    logger.Error("не удалось проверить отзыв токена", ...)  // Только логирует!
} else if revoked {
    c.AbortWithStatusJSON(http.StatusUnauthorized, ...)
    return
}
```

### Стратегия эксплуатации

**Фаза 1 - Рост файла**: Генерируем множество OneTime VIP токенов и используем их для увеличения `/var/lib/revocation/revoked.txt`:
- Каждое использование VIP токена запускает автоматический отзыв (after-hook middleware)
- Файл растёт ~1 запись на токен, замедляя будущие сканирования `IsRevoked()`

**Фаза 2 - Запуск дедлока**: Когда файл становится достаточно большим, что сканирование превышает таймаут 50ms, следующий вызов `CheckRevoked()` заблокирует mutex

**Фаза 3 - Эксплуатация**: Выдаём свежий VIP токен → доступ к курсу 4 → извлекаем slug урока 16 → доступ к уроку с отозванным токеном (проходит из-за fail-open)

- TTL OneTime токена: 10 секунд (достаточное окно)

### Извлечение флага

После блокировки сервиса отзыва:
1. Запрашиваем свежий OneTime VIP токен для урока 15 → получаем токен B
2. Используем B для GET `/api/courses/4` → извлекаем slug урока 16 из ответа
3. GET `/api/courses/4/lessons/access/<slug16>` с B → возвращает флаг
4. B технически отозван после шага 2, но заблокированный чекер не может это проверить
