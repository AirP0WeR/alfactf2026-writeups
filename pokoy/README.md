## Решение

### 1. Первоначальная разведка — что внутри архива

Архив `peace_6ce0fda.tar.gz` содержит единственный файл `peace.v` — Yosys-flattened Verilog-нетлист (~58 000 строк, 2 MB) синтезированной FPGA-логики устройства. Никакого C/Python исходника, только уже оптимизированная вентильная схема.

Цель: устройство принимает 64 байта ASCII (флаг) с 4×4 клавиатуры, прогоняет их через встроенный шифр и отображает 64-байтовый шифротекст на OLED 128×64. Нужно инвертировать шифр, получив исходный ввод.

### 2. Идентификация шифра

В `peace.v` среди тысяч вентилей читаемые идентификаторы:

- `_M_00000_` … `_M_00015_` — 16 таблиц по 256 записей × 8 бит (финальные S-box'ы).
- `_M_00016_` … `_M_00159_` — 144 таблицы по 256 × 32 бит.

Структура: 9×16 + 16 = ровно AES-128 в T-table форме:
- 9 полных раундов (каждый использует 4 T-таблицы на колонку × 4 колонки = 16 таблиц на раунд).
- Финальный раунд без MixColumns — только 16 S-box'ов.

Раундовые ключи «запечены» внутрь таблиц (white-box-light AES): стандартных KeyExpansion-констант в нетлисте нет, но XOR с ключом поглощён в каждую запись таблицы.

`solve/extract.py` — парсер Verilog'а, дампит все таблицы в `solve/aes_data.json` (~451 KB).

### 3. Восстановление раундовых ключей из таблиц

Для каждой T-таблицы перебираем 256 кандидатов `key_in ∈ [0, 255]`:

```
T[x]  ==  AES_T_standard_b[AES_SBox(x ^ key_in)] ^ const_mask
```

где `b` — номер байта (0–3) внутри 32-битного слова T-таблицы. Если формула выполняется для всех 256 входов — кандидат является раундовым ключом позиции `(round, row, col)`. Для финальных S-box'ов аналогично, но без MixColumns:

```
Sbox_table[x]  ==  AES_SBox(x ^ key_in) ^ key_out
```

Это даёт `(key_in, key_out)` для каждого из 16 байт последнего раунда. Реализация — `solve/solve.py`, функция `extract_table_info()` (строки 77–106).

### 4. Перестановки байт

FPGA вводит два слоя переупорядочивания, закодированных в проводниках нетлиста:

- `INPUT_POS` — порядок, в котором набранные байты пишутся во входной регистр AES.
- `OUTPUT_POS` / `Q_OF_P` / `QIN_OF_P` / `P_BY_COL` — порядок, в котором байты выходного регистра передаются на дисплей.

Обе таблицы восстановлены трассировкой имён сигналов в `peace.v` и зафиксированы в `solve/solve.py` (строки 41–47).

### 5. Процедура расшифрования

Функции `decrypt_internal_block` и `solve_display_block` в `solve/solve.py`:

1. Реверсировать 16-байтовый блок с дисплея (порядок внутри 128-битного регистра обратный).
2. Применить обратные финальные S-box'ы с извлечёнными `(key_in, key_out)`.
3. Для раундов 8→0 по каждой колонке: XOR с маской T-таблицы, обратный MixColumns, обратные раундовые S-box'ы.
4. Реверсировать результирующий plaintext (буфер ввода хранится в обратном порядке относительно канонического AES-состояния).

Шифротекст разбивается на 4 блока по 16 байт; каждый блок обрабатывается независимо.

### 6. Верификация — прямое шифрование

`solve/forward_aes.py` строит зеркальное зашифрование через те же извлечённые таблицы и повторно шифрует восстановленный флаг. Вывод совпадает с целевым шифротекстом побайтово. Полный round-trip (encrypt → decrypt → plaintext) также проходит корректно.
