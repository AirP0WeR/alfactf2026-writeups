# Покой — Reverse + Hardware (1000pt, 0 solves)

## TL;DR
Устройство — 4×4 keypad + 128×64 OLED-дисплей, описанный в `peace.v` (Yosys-flatten,
~58k строк). Дисплей рендерит 64 байта как **8 строк × 16 hex-цифр** (8×8 шрифт).

В прошивке найдены **16 финальных 8-bit S-boxes** (`_M_00000_`…`_M_00015_`) и
**144 32-bit T-tables** (`_M_00016_`…`_M_00159_`). Структура `16 × 9 + 16` = ровно
**AES-128** в T-table стиле (9 полных раундов + финальный без MixColumns).

Цель: ciphertext-матрица 64 байт известна (из условия), нужно восстановить keypress-последовательность
(плейнтекст 4×16 байт), которая после симуляции даёт эту матрицу. Затем декодировать
её обратно в строку флага `alfa{...}`.

## Целевые 64 байта (CT)
```
71 80 dd 08 6c 43 37 6b 3c a8 e2 68 6d 0a 97 9b
f8 23 b9 66 6a b9 92 d3 cd ad 80 cf e8 5a ed 1f
c5 c4 6c 60 c0 a3 a5 c9 11 a1 d1 98 f1 83 53 95
53 e4 b9 7c 16 5f c4 3b a3 f5 1e 12 53 1f ea ca
```
Это 4 AES-блока по 16 байт. Порядок байт внутри блока — TBD (см. T4).

## Известный hardware-протокол (peace.py)
- Тактовая HZ = 1000.
- HOLD_CYCLES = 110, GAP_CYCLES = 110, COMMIT_CYCLES = 598 (HZ/2 + 4 + 64 + 30),
  INITIAL_SETTLE = 98.
- Keypad: активный low по колонкам ТОЛЬКО когда строка стробится (rows = output, cols = input).
- KEYMAP: `1234`/`5678`/`90c#` (или с буквами A-D), индекс 0..15.
- "Repeat-press window" 0.4s — несколько нажатий одной клавиши до коммита склеиваются
  (классический T9 multi-tap).
- После коммита запускается внутренний пайплайн (~498 тактов), потом дисплей валиден.

## Архитектурные гипотезы

### H1 (надежда): "Vanilla T-table AES" с baked round keys
T_r,b[x] = MixColumns_b(AES_SBox(x ⊕ k_r,b)) — обычный Rijndael, где раундовый ключ
уже зашит в T-таблицу. Финальные S-boxes = AES_SBox(x ⊕ k_10,b).
**Если так:** ключ извлекается перебором 256 вариантов на байт (16×10 = 160 переборов).
Финальный шаг — `AES_decrypt(CT, recovered_key)` поблочно.

### H2: AES + аффинные внешние кодировки
T_r,b[x] = P_out(MixColumns(SBox(P_in(x) ⊕ k_r,b))). Ломается через аффинный криптоанализ
таблиц или DDT/LAT-эквивалентность к AES_SBox.

### H3 (худший случай): полный white-box AES (Chow et al.)
Нелинейные encodings, нужен BGE/MGH-attack. Известные имплементации (есть Python
библиотеки), но дольше.

**Быстрая проверка H1:** sboxes[0][:16] не совпадает с AES_SBox[:16] и не выглядит
константным XOR'ом (см. probe-вывод). Но этого мало — нужно проверить аффинную
эквивалентность через DDT (T3). Если sbox — пермутация AES_SBox под 2 GF(2)^8-аффинными
картами, то это white-box-light.

## План (детально)

### T1 — оракул keys → 64 байта CT (BLOCKER на T2/T4)
- Реанимировать `oracle.py` и `decode2.py` (там парсер дисплея в hex).
- API: `simulate(keys: str) -> bytes` (возвращает 64 байта).
- Sanity: пустой ввод → запомнить CT_0; ввод "1" → CT_1.

### T2 — keypad → байт буфера M_00160
- Простой test: одиночное нажатие "1" → какие байты M_00160 ненулевые? Какие значения?
- Двойное "11" — другая позиция или замена? (T9 multitap → одно "11" даст другой код).
- Нажать прозрачные шесть символов "1afle{}" (через возможный multitap) и снять M_00160.
- Цель: bijection symbol→byte_value, plus длина по символу (1, 2, 3 нажатия?).

### T3 — структура шифра
- Извлечь все 16 sboxes + 144 ttables (готово).
- Для каждой sbox_i: построить DDT, проверить max diff = 4 (как у AES_SBox).
- Найти аффинную пару (A, b, c, d) такую что sbox_i(x) = A·AES_SBox(B·x ⊕ c) ⊕ d.
  Если existует — H2/H1, иначе H3.
- Для T-таблиц: проверить, что T[x] это копия 4-байтового AES T0/T1/T2/T3 с XOR константы.

### T4 — байт-маппинг 4 блоков
- Симулятор: для i=0..63 поставить M_00160[i] = 0x01 (через ввод который даст такой байт),
  снять CT, посмотреть в каком блоке/байте поменялись данные.
- Получить permutation π: PT_index ↔ (block_idx, byte_idx_in_block).

### T5 — извлечение AES ключа
- Если H1: перебираем k_r,b ∈ 0..255 так что T_r,b[x] = AES_T_b[AES_SBox(x ⊕ k_r,b)].
- Сложить раундовые ключи k_1..k_9, добавить k_10 из финальных sbox. Проверить
  согласованность через AES key schedule (round keys должны выводиться друг из друга).
- Готовый код: использовать `cryptography` или собственный AES для inverse.

### T6 — расшифровать 4 блока
- Прямой AES-128-decrypt каждого 16-байтового блока CT.
- Получить 64 байта плейнтекста = последовательность keypad-байтов.

### T7 — байты → клавиши → флаг
- Применить обратную таблицу T2.
- Соединить символы, дополнив `alfa{` префиксом и `}` суффиксом если ожидается.
- Валидировать через `scripts/flag.sh`.
